# CHAMP-PM Full Codebase Audit

**Date:** 2026-06-12  
**Auditor:** Fable 5 (3 parallel subagents — backend, frontend, infra)  
**Scope:** All 82 API routes, all React pages/components, all migrations, all config  
**Live probes:** Non-destructive checks against https://champ-pm.app  

---

## Executive Summary

The codebase is in better-than-average shape for a small-team internal app. The domain invariants (salary append-only, fringe immutability, 0.317 F&A, PoP ceilings) are consistently enforced in code. JWT verification is real (not trust-the-claim). SQL injection is almost entirely absent — 200+ parameterized queries with one lone exception. Secrets have never touched git.

**However**, there are issues that need immediate attention:

1. **SQL injection** in `program-schedule/index.js` — one raw string interpolation in an otherwise clean codebase
2. **Employee HR data** (names, termination dates, salaries, UINs) exposed in: the public JS bundle, git history, and API source code
3. **Production source maps** served publicly at `https://champ-pm.app/assets/*.js.map` — full commented source readable by anyone
4. **Clerk SDK in known-vulnerable range** — auth bypass CVE with a non-breaking fix available (`npm audit fix`)
5. **PRIDE sync endpoint is completely broken** (middleware 401s it before the handler's own auth runs) — and when it's fixed, it needs timing-safe token comparison and salary validation

## CRITICAL FINDINGS

### C1. SQL Injection in `program-schedule/index.js`
**File:** `functions/api/program-schedule/index.js` (~line 15)

```js
let grantWhere = grantStatus === 'all' ? '' : `WHERE g.status = '${grantStatus}'`;
```

`grant_status` comes from `url.searchParams.get('grant_status')` and is interpolated directly into SQL. Every other query in the codebase uses `.bind()` parameterization — this is the lone exception. An authenticated admin can inject UNION-based queries to dump any table (users, salaries, clerk_ids).

**Fix (5 minutes):**
```js
const ALLOWED = ['active', 'closed', 'pending', 'all'];
if (!ALLOWED.includes(grantStatus)) return json({ error: 'invalid grant_status' }, 400);
// alternatively, just parameterize: WHERE g.status = ?
```

### C2. Employee PII and Planned Terminations Hard-Coded in Public JS Bundle
**File:** `src/pages/admin/StaffPlans.jsx` (~lines 11–49)

```js
const TERMINATIONS = {
  mjr: '2026-04-30',    // Mary Richardson — no appointments past this date
  dianad: '2026-04-30', // Diana Davisson
  lkumar: '2026-04-01', // Love Kumar — no new appointments
};
const STAFF_NAME_MAP = { /* 28 staff: netID → full name */ };
```

This compiles into the public JS bundle served at `/assets/index-*.js`, accessible without authentication. **Planned termination dates for named employees** are exposed to anyone who can fetch the URL — which is everyone. The TERMINATIONS map is also sent to the API as client-supplied data (server should never trust the client for this).

**Fix:**
1. Move `TERMINATIONS` into D1 (e.g., `separation_date` column on `users`) — have optimizer read it server-side; never accept termination dates from client
2. Replace `STAFF_NAME_MAP` with `user_name` already returned by the staff-plans endpoints (the fallback already does this)
3. Redeploy immediately to purge from CDN; invalidate Cloudflare cache

### C3. Clerk SDK in Known-Vulnerable Range — Auth Bypass CVEs
**File:** `package.json` (`@clerk/clerk-react 5.61.3`, `@clerk/shared`)

`npm audit` reports:
- **Critical (GHSA-vqx2-fgx2-5wq9):** Middleware-based route protection bypass in `@clerk/shared`
- **High (GHSA-w24r-5266-9c3c):** Authorization bypass in `@clerk/clerk-react` 5.9.0–5.61.5
- **High:** `js-cookie` cookie-attribute injection via prototype hijack

CHAMP-PM does its own JWKS JWT verification (which insulates the API), but staying in a known-vulnerable range of an auth SDK is unacceptable posture.

**Fix (15 minutes, non-breaking):** `npm audit fix` and redeploy. Re-run `npm audit` to confirm clear.

### C4. Production Source Maps Publicly Served

Full original source — every page, API call pattern, business logic, internal naming, salary/equity internals — is downloadable by anyone who can reach the site. Combined with C2, this exposes HR data in human-readable commented form.

### C5. Migrations Cannot Reproduce Production — No D1 Backup Story
**Files:** `migrations/` directory, `functions/api/pride/sync.js`

Multiple compounding problems:
1. `users.end_date` (used by PRIDE sync) **does not exist in any migration** — it was added directly to production D1. A replay from migrations crashes the deployed code.
2. Seed migrations reference `grant_id = 7, 8, 11–14, 15–17` but no migration creates those grants (added ad hoc). Fresh DB → FK violations.
3. `0023_staff_plans_priority.sql` explicitly notes *"Already applied directly to D1"* — confirming the ad-hoc-first workflow. Wrangler replaying it would `ALTER TABLE` error on duplicate column.
4. The 17× `0016_timesheet_import_*` shards plus 4 other duplicate-prefixed files (0002, 0003, 0019) make ordering non-obvious.

**If D1 is lost, migrations cannot restore production.** D1 time-travel is 30 days.

**Fix:**
- Take a full export NOW: `npx wrangler d1 export champ-pm --remote --output=backup-$(date +%F).sql`
- Write catch-up migration `0025_schema_truth.sql` with `ALTER TABLE users ADD COLUMN end_date TEXT;` plus grants seed documentation
- Never use ad-hoc `d1 execute` for schema changes going forward — write migration first

## HIGH FINDINGS

### H1. Email-Claim Account Linking Can Hijack User Identity
**File:** `functions/_middleware.js` (~lines 118–127)

When a Clerk `sub` isn't found in D1, the middleware falls back to email match and writes the new Clerk ID onto that user row — without checking `email_verified` or that `clerk_id IS NULL`. A second Clerk account with the same email overwrites the legitimate user's D1 identity.

**Fix:** Require `payload.email_verified === true`, only link `WHERE clerk_id IS NULL`, log all link events.

---

### H2. PRIDE Sync Endpoint Is Completely Broken (Verified Live)
**Files:** `functions/_middleware.js`, `functions/api/pride/sync.js`

Global middleware requires a Clerk JWT on all `/api/*`. The PRIDE bookmarklet sends a shared secret (`PRIDE_SYNC_TOKEN`), not a JWT. Live probe returns the middleware's "Invalid JWT format" — the handler's own auth and all its salary-insert logic are dead code.

**Fix:** Add a narrow carve-out in middleware (`if (url.pathname === '/api/pride/sync') return next()`) — but only after also fixing H3 (timing-safe compare) and validating the salary payload. Do not broaden the carve-out.

---

### H3. PRIDE Token Comparison Is Not Timing-Safe + Unvalidated Salary Payload
**File:** `functions/api/pride/sync.js` (~line 90)

```js
if (!token || token !== expectedToken) { ... }
```

Once the middleware carve-out is in place (H2), this becomes the only gate on an unauthenticated endpoint that inserts salary records. The `!==` comparison short-circuits. Additionally, `salary` from the payload is inserted unvalidated — a string or `1e12` is accepted; `Math.abs(current - salary)` with a non-numeric value is `NaN`.

**Fix:**
```js
const enc = new TextEncoder();
const a = enc.encode(token), b = enc.encode(expectedToken);
// crypto.subtle.timingSafeEqual on ArrayBuffers is available in Workers
```
Also: validate `salary` is a finite positive number in a sane band (e.g. $20k–$400k) before insert.

---

### H4. AI Goals Endpoint Applies Unvalidated LLM Output as Financial Constraints
**File:** `functions/api/staff-plans/ai-goals.js` (~lines 195–215)

Claude's JSON response is `JSON.parse`d and fed directly into the LP optimizer with zero schema validation. A negative cap, `NaN` multiplier, or prompt-injected exclusion flows straight into persisted `staff_plan_scenario_rows`. Grant names and staff names from the DB are interpolated into the prompt — anyone who can name a grant/user can embed instructions. The result deletes and replaces non-pinned scenario rows before anyone reviews.

**Fix:** Strict schema validation on all AI overrides (range-check every numeric value, cross-reference account strings and user IDs against DB). Treat AI runs as a preview requiring an explicit apply step.

---

### H5. JWT Verification Gaps in Middleware
**File:** `functions/_middleware.js` (~lines 49–86)

Signature verification is real (RS256, JWKS, exp checked ✅), but:
- No `iss` (issuer) validation
- No `azp`/`aud` validation (token reuse from other Clerk apps)
- `header.alg` is never checked — pin `alg === 'RS256'`
- On JWKS `kid` miss, code throws instead of re-fetching — after a key rotation, all logins break for up to the cache TTL
- Role from JWT claims only — prefer `dbUser.role` from D1 as authoritative

### H6. `xlsx` 0.18.5 — Prototype Pollution + ReDoS, No Fix on npm
**File:** `package.json`

CVE-2023-30533 (prototype pollution) and CVE-2024-22363 (ReDoS). npm version is frozen at 0.18.5. The app parses **user-uploaded spreadsheets** in StaffPlans and QuarterlyReports — exactly the attack path.

**Fix:** `npm i https://cdn.sheetjs.com/xlsx-0.20.3/xlsx-0.20.3.tgz` (API-compatible)

---

### H7. No Code Splitting — Admin Code Ships to Every User
**File:** `src/App.jsx` (all 30+ pages eagerly imported), `vite.config.js` (no `manualChunks`)

Staff users download all 26 admin pages, recharts (~450 KB), xlsx (~430 KB), and Gantt machinery. No `React.lazy` / `Suspense` anywhere. This increases the PII exposure surface (C2's termination data ships to all users) and hurts first-load.

**Fix:** `React.lazy(() => import(...))` + `<Suspense>` at minimum for admin vs. staff route groups, recharts-heavy pages, and xlsx-using pages.

---

### H8. No ESLint — Bugs and Hook Violations Go Undetected
**Files:** repo root (no `.eslintrc*`)

No ESLint config means `react-hooks/exhaustive-deps`, undefined-variable, and unused-import checks never run. Already caused C1-frontend (`api.delete` doesn't exist). Stale-closure deps exist in `Staff.jsx`, `Reports.jsx`, `ProjectDetail.jsx`.

**Fix:** Add `eslint` + `eslint-plugin-react` + `eslint-plugin-react-hooks`; run in CI before deploy.

---

### H9. No Security Headers — No CSP, X-Frame-Options, HSTS
**Files:** `public/` (no `_headers` file), `index.html`

No `Content-Security-Policy`, `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`. For an app displaying salary/HR data this matters.

**Fix:** Add `public/_headers`:
```
/*
  X-Frame-Options: DENY
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Content-Security-Policy: default-src 'self'; script-src 'self' https://*.clerk.accounts.dev; connect-src 'self' https://*.clerk.accounts.dev; img-src 'self' data: https://img.clerk.com; style-src 'self' 'unsafe-inline'; frame-ancestors 'none'
```
(Tune Clerk hostnames; start with `Content-Security-Policy-Report-Only`)

## MEDIUM FINDINGS (Selected — See Full Reports)

### Backend Mediums
- **M-B1: CRM read endpoints skip auth** — any authenticated user (even unprovisioned in D1) can dump all contact/org/interaction data
- **M-B2: Unprovisioned Clerk users can write timesheet data** — if `dbUser` is null, `userId` becomes Clerk's `sub` string
- **M-B3: Import-replace wipes table BEFORE validating payload** — `replace:true` with bad `rows` guarantees data loss
- **M-B4: Resubmitting a week destroys approval audit trail** — `ON CONFLICT DO UPDATE SET reviewed_by=NULL` lets staff clobber approved records
- **M-B5: Stack traces leaked to clients** — optimizer catch returns `{ error: String(err), stack: err?.stack }` to client; many routes return raw `err.message` with schema details
- **M-B6: Date strings never format-validated** — PoP ceiling check uses lexicographic comparison that breaks for non-ISO dates; `12/31/2035` bypasses the ceiling
- **M-B7: Numeric inputs not coerced/range-checked** — SQLite dynamic typing silently accepts strings in REAL columns; `salary / 2080` → `NaN` propagates into budget rollups
- **M-B8: CSV formula injection + broken quoting** in timesheet report export — names from PRIDE/import can contain `=` or `"`
- **M-B9: CORS preflight returns `Access-Control-Allow-Origin: *`** while actual responses carry nothing — inconsistent and widens if cookies ever added
- **M-B10: University UINs and full employee names hard-coded in source** (`pride/sync.js` and `appointments/import.js`) — maintenance burden and privacy risk

### Frontend Mediums
- **M-F1: Frontend-only RBAC** — protected routes gate on Clerk public metadata; must be verified backend-side on every route (cross-check with backend audit)
- **M-F2: Broken onboarding link** — `/staff/timesheet` doesn't exist; new staff finish onboarding and hit blank page
- **M-F3: `BURN_MULTIPLIER = 1.451 * 1.317` hardcoded** in StaffPlans visualization while the rest of the app treats fringe as DB-driven
- **M-F4: StaffPlans.jsx is 2,011 lines** with duplicate Modal, Field, formatters, dead `parseCSV`, and 5 tabs that should be split
- **M-F5: No memoization** across chart/table-heavy renders — Equity, StaffPlans, Dashboard rebuild expensive aggregations on every state change
- **M-F6: Missing React keys on fragments**; 17 instances of `key={index}` on dynamic lists
- **M-F7: Modal accessibility** — no Escape key, no focus trap, no `role="dialog"`/`aria-modal`/`aria-labelledby`
- **M-F8: Dead/stub code in production** — `updateStatus()` is a no-op toast, `parseCSV` unused, hidden empty div in Staff.jsx
- **M-F9: PRIDE bookmarklet stores sync token in third-party localStorage** — any XSS on `pride.prairie.illinois.edu` exfiltrates it

### Infra Mediums
- **M-I1: Remaining npm audit findings** — `react-router` open redirect (runtime), `lodash` prototype pollution (via recharts), `esbuild` dev-server forgery (dev-time only)
- **M-I2: Duplicate migration prefixes** (0002, 0003, 0016, 0019) — ordering correctness relies on accidental alphabetics
- **M-I3: `ON DELETE CASCADE` from contacts** destroys all interaction history for that contact — no soft-delete
- **M-I4: No DB-level append-only enforcement** on `salary_records`/`fringe_rates` — convention only; a buggy endpoint or future developer can UPDATE/DELETE
- **M-I5: Missing foreign keys on post-0001 tables** — `salary_records.user_id`, `grant_fa_rates.grant_id`, etc. allow silent orphans
- **M-I6: No CHECK constraints** on `allocation_pct` (0–100), `hours` (≥0), `rate` (<1) — a unit-mistake on fringe (0.451 vs 45.1) inflates all cost calculations 100×
- **M-I7: `ON DELETE CASCADE` from projects → schedule phases/milestones/scenarios** — deleting a project wipes all schedule history

## LOW FINDINGS (Summary)

**Backend:** Dead code in `balances/[id].js` sync delegation; destructive `DELETE /api/staff-plans/appointments` with no friction; missing try/catch on JSON body parsing in most routes; N+1 / sequential query patterns in recalculate/AI-goals/pride-sync; no pagination on several list endpoints; no rate limiting (especially `ai-goals` which burns Anthropic tokens per call); equity snapshots store client-computed values instead of recomputing server-side; F&A and fringe literals appear in 5+ files instead of being centralized.

**Frontend:** Waterfall fetch in Staff detail modal (fetches entire org equity to filter for one user); `useApi` has no caching so staff/grants lookups are re-fetched on every navigation; `window.confirm`/`prompt` used for destructive confirmations (18 instances); toast container not `aria-live`; `hourly` role exists in Staff form but causes a redirect loop (not in `ProtectedRoute` allowlist); timezone-fragile date math mixed UTC/local across files.

**Infra:** CORS wildcard on preflight; `compatibility_date = "2024-01-01"` is 2.5 years stale; `datetime("now")` with double-quotes (should be single) in migration 0022; inconsistent `updated_at` defaults in CRM tables; missing indexes on interaction/schedule/CRM join paths; favicon 404 on every page load; PRIDE sync hard-codes `0.451` fringe and `'surs'` type for all employees (breaks for non-SURS hires); `'astillwell'` UIN mapping doesn't match seeded user ID `'ashlynn'` — silently inserts salary for nonexistent user.

## Things Done Well

These are genuine strengths worth preserving:

1. **Parameterized SQL is near-universal** — 200+ D1 queries reviewed; one injection (C1) in an otherwise clean codebase
2. **Real JWT signature verification** — RS256 against Clerk JWKS with WebCrypto; not trust-the-claim
3. **Domain invariants enforced in code** — salary append-only, fringe immutability, 0.317 F&A consistency, PoP ceilings with 409s, referential guards before deletes
4. **Consistent admin gating** — `requireAdmin()` correctly applied across ~75 of 82 routes; self-scoped exceptions (`tasks/my`, `timesheets/submit`, `staff/me`) are intentional and correct
5. **Secrets discipline** — `CLERK_SECRET_KEY`, `ANTHROPIC_API_KEY`, `PRIDE_SYNC_TOKEN` only in Pages secrets; `.env` never committed (verified in git log)
6. **No XSS sinks in React** — zero `dangerouslySetInnerHTML`, `innerHTML`, or `eval` across ~12k lines of JSX
7. **Timesheet week locking** — submitted/approved weeks block staff edits on both POST and DELETE
8. **AI endpoint degrades gracefully** — Claude failures return 502 without leaking upstream error; unparseable JSON falls back to standard optimization
9. **Import pipeline is thoughtful** — encoding fallback, format auto-detect, 400-row D1 batch chunking, idempotent duplicate-skip
10. **Bookmarklet XSS hygiene** — `textContent` only, no injection sink, token never embedded in distributed file, CORS origin-pinned to PRIDE host
